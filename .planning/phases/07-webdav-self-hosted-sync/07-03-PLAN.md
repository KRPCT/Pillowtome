<objective>
Land the SYNC-04 file plane: per-book opt-in upload of book bytes to the user's self-hosted WebDAV root, and on-demand download on the peer device. Upload is a threshold state machine ‚Äî below 10MB a single streaming PUT (`Body::from(tokio::fs::File)` + `If-None-Match: *`, never reqwest_dav `put(Vec<u8>)`); at/above 10MB a server-capability branch: Nextcloud chunked upload v2 (MKCOL upload dir, zero-padded integer chunks, Destination on every request, OC-Total-Length quota precheck, MOVE .file assembly, 423/504 backoff, 24h expiry) versus generic RFC-4918 servers, which get a streaming whole-file PUT with restart-from-zero retry. This branch **adopts research Q1 as an explicit correction to D-101**: RFC 4918 has no server-side assembly, so a "chunk PUT sequence" is impossible there ‚Äî Nextcloud chunk v2 is a private protocol and is the only resume-capable path in v1. Download probes `Accept-Ranges`, streams ranged chunks into `<file>.part` with resume, then enforces blake3 == work_id (work_id IS the blake3 hex ‚Äî `work_id_from_hash` is identity, `src-tauri/src/commands.rs:110`) before renaming into app-owned storage and handing the path to the existing ingest entry point (D-100).

Purpose: Books are the big payload ‚Äî the state plane (07-02) syncs progress/annotations/catalog every session, but a 300MB PDF moves once, on user opt-in (D-98), and must survive proxy body limits, 504s, Nutstore throttling, and hostile/corrupt servers without ever touching memory wholesale or the IPC channel (D-06).

Output: `core/src/sync/fileplane.rs` pure planner (off-device tested), `src-tauri/src/sync/fileplane.rs` upload/download state machines, `src-tauri/tests/sync_fileplane.rs` wiremock integration tests, and two NEW `#[tauri::command]` fns created by this plan in `src-tauri/src/sync/commands.rs` (`sync_set_file_sync`, `sync_download_book`) plus their registration in `src-tauri/src/lib.rs`'s `tauri::generate_handler!`. No new runtime dependency in either crate (D-13).
</objective>

## Deviation Note (deliberate, bounded)

The single-PUT upload path (books below 10MB) writes DIRECTLY to the final remote name (`books/‰ΩúËÄÖ - ‰π¶Âêç.ext`) ‚Äî no tmp-name + MOVE ‚Äî a deliberate deviation from the state plane's tmp+MOVE atomic-rename rule (Pitfall 5). Bounded because: (1) `If-None-Match: *` guards first creation, so a concurrent/partial write can never silently clobber an existing remote book; (2) a failed PUT simply retries the whole stream ‚Äî restart-from-zero is cheap below 10MB; (3) the download side's blake3 == work_id gate means a peer can NEVER accept a partial/corrupt file into its library ‚Äî the worst case of an interrupted upload is a corrupt remote object that fails verification and gets re-uploaded, never a poisoned library. The chunked path already assembles off to the side (upload dir + MOVE `.file`), and the state plane keeps tmp+MOVE unchanged.
